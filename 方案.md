# GraphResearcher 架构完善方案

这套方案的核心不是把传统 DeepResearch 的 TODO list 和 evidence store 简单换成图数据库，而是把“层级式 Task 规划、搜索过程、证据链条、冲突处理、报告写作”统一建模为可查询、可评分、可回溯的知识图谱。Task 是图谱中的基本规划单元，query 是 Task 的检索表达或搜索问题文本；每个 Task 都维护自己的搜索结果子图，包括 Document、Evidence、Claim、Source 等节点。现有 `src/prompts/researcher/zh/qwen3.txt` 的 workflow 已经提供了成熟的编排范式：主控 Agent 负责规划、调度、反思和最终验收；Searcher 负责检索和证据收集；Reporter 负责证据绑定写作。GraphResearcher 可以在此基础上把 TODO 状态和 evidence_store 升级为 Graph Manager 管理的 Research Graph。

## 1. 总体架构

GraphResearcher 由一个主控 Agent 和三个 Sub-Agent 组成：

- **Orchestrator**：负责理解用户需求、创建初始 Task 层级图谱、调度 Searcher、分析每个 Task 的搜索结果子图充分性、聚合同层 Task 判断是否横向扩展、判断单个 Task 是否纵向深入、触发 Reporter，并对最终报告进行验收。
- **Searcher**：负责调用网页搜索、学术搜索、GitHub 搜索或其他垂类搜索工具；读取网页内容；抽取 claim、evidence、source metadata；将结构化结果交给 Orchestrator 或直接写入 Graph Manager。
- **Reporter**：只基于 Research Graph 写作，不做大规模外部检索；先读取图谱生成证据绑定大纲，再按章节写作，最后汇总报告并保留引用映射。
- **Verifier**：负责异常和质量问题处理，包括重复 Task/query、证据冲突、来源低质、引用缺失、章节结论与证据不一致、搜索失败重试后仍无结果等。

两个基础组件：

- **Graph Manager**：管理 Research Graph 的 Toolkit，承载 task、document、claim、evidence、source、conflict、analysis、section 等节点及边。
- **Middleware**：系统护栏，处理工具调用失败、schema 校验、重试、预算限制、日期/时效性提醒、图写入事务一致性和输出格式修复。

## 3. Orchestrator Workflow

### Stage 1：任务理解与图谱初始化
1. 解析用户意图，识别研究目标、报告语言、目标受众、时间窗口、输出要求、禁止范围。
2. 生成一组同等粒度的初始 `Query` 节点。
3. 生成 `checklist` 或 `acceptance_criteria`，用于后续判断研究是否可以收敛。
4. 初始化图谱，调用 `Add_Node` 将初始 `Query` 节点入图。

### Stage 2：图谱驱动的调研规划与证据收集
循环执行以下流程：
1. **选择待执行 query**：从图谱中选出 `pending`、`needs_horizontal` 或 `needs_vertical` 的高优先级 query。可并行时每轮选择 2-4 个。
2. **调用 Searcher**：为每个 query 构造检索任务，要求 Searcher 检索并将研究结果入图。
3. **子图充分性分析**：调用工具对每个 query 的 ego-subgraph 进行评分，判断当前证据是否足以支撑该 query。
4. **规划下一步方向**：
   - 如果当前层级 query 所属维度不完整，进行横向扩展。
   - 如果 query 内部已有证据但关键机制、原因、数据口径或冲突未解决，进行该分支的纵向深入。
   - 如果证据充分且冲突可解释，标记为 `sufficient`。
   - 如果无法获得可靠证据，标记为 `blocked`，交给 Verifier 判断是否更换搜索策略或在报告中披露限制。
5. **记录中间分析**：当出现阶段性结论、框架、取舍、冲突处理方案时，写入 `Analysis` 节点，供 Reporter 复用。

### Stage 3：图谱绑定的报告生成
1. Orchestrator 调用 Graph Manager 导出报告上下文，包括 query tree、claim-evidence map、conflict list、analysis notes
2. Reporter 基于完整图谱生成证据绑定大纲。每个章节必须显式绑定对应 query、claim 和 evidence。
3. Reporter 按章节写作。章节写作前检查证据覆盖和冲突；章节写作后将 `ReportSection` 节点与使用的 evidence/claim 建立 `USES` 边。
4. Reporter 汇总最终报告，保留正文编号引用和统一参考文献。
5. Orchestrator 调用 Verifier 对最终报告验收，包括事实-证据一致性、引用完整性、章节逻辑一致性、冲突是否披露。

### 横向扩展说明

横向扩展用于补齐同一层级的研究维度。触发条件：
- 当前 query 只覆盖了问题空间的一部分，根目标仍缺少平行维度。
- 多个 evidence 暗示存在重要相邻主题，但初始 query 没覆盖。
生成规则：
- 新 query 与当前 query 共享同一 parent。
- 粒度必须与当前 query 保持一致。
- 避免重复已有 query，可先做语义去重。
- 每次横向扩展优先生成 1-3 个高价值 query。

## 纵向深入说明

纵向深入用于沿着当前 query 的证据链继续追问。触发条件：
- 已找到初步结论，但缺少机制、原因、约束、数据口径、反例或适用边界。
- Claim 有证据支持但可信度不足，需要一手来源或交叉验证。
- 多个来源冲突，需要追问冲突原因。
- 证据出现关键实体、技术路径、指标、案例，值得进一步深挖。
生成规则：
- 新 query 作为当前 query 的 child。
- 问题更具体，必须继承父 query 的上下文。
- 每个子 query 应明确要验证哪个 claim、解决哪个 gap 或解释哪个 conflict。

### 停止条件与收敛判断
1. 预设定的轮次
2. 图谱状态：
  1. 所有高优先级 query 节点均已完成且分数达到阈值（调研相对充分）且无致命冲突
  2. 低优先级 query 节点的剩余缺口不影响核心结论
  3. 新一轮搜索结果的 novelty 很低，继续搜索收益不足，需如实记录并反馈在报告中

## 7. Verifier 职责

Verifier 不负责常规检索和写作，而负责异常判断与质量验收：

- **重复问题处理**：识别语义重复 query、同源重复 source、重复 claim。
- **冲突处理**：判断冲突是事实矛盾、时间差异、统计口径差异、观点差异还是来源低质。
- **证据质量检查**：标记低可信来源、过时来源、缺少发布时间的来源。
- **引用检查**：检查正文引用是否全部映射到 source，编号是否唯一且连续。
- **报告一致性检查**：章节结论是否与图谱 claim 一致，是否引入未入图事实。
- **失败恢复建议**：当 Searcher 连续失败或图谱写入异常时，给出重试、降级或停止建议。

## 8. Middleware 护栏

Middleware 应覆盖以下系统级问题：

- 工具失败重试：网络异常、搜索 API 超时、图写入失败。
- Schema 校验：Searcher/Reporter/Graph Manager 的输入输出必须符合结构化 schema。
- 图事务一致性：写入 source/evidence/claim/edge 时保证原子性，失败则回滚或补偿。
- 预算与轮次控制：限制最大搜索轮次、并发数、单轮 query 数和总证据量。
- 时效性护栏：对日期敏感任务强制记录发布日期和检索日期。
- 幻觉护栏：禁止 Reporter 使用未入图的事实性陈述。
- 引用护栏：最终报告只能使用图谱中的 source URL 生成参考文献。
